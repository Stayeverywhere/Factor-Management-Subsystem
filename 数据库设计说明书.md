## 1. 引言

### 1.1 编写目的
本文档旨在描述因子管理子系统（Factor Management Subsystem）的数据库设计方案，明确数据表结构、字段定义、索引策略、数据关系及初始数据，为开发、测试及运维人员提供统一的数据规范参考。

### 1.2 适用范围
本文档适用于因子管理子系统的所有模块，包括：
- 认证与权限管理（Auth）
- 因子分类与基础因子管理（Factor Category / Base Factor）
- 衍生因子与风格因子管理（Derivative Factor / Style Factor）
- 基金数据同步（akshare 数据同步）
- 交易银子账户管理（Trade）

### 1.3 数据库环境

| 项目 | 说明 |
|------|------|
| 数据库类型 | KingbaseES（兼容 PostgreSQL 15+） |
| Schema | `biz_factor` |
| 字符集 | UTF-8 |
| 时区 | Asia/Shanghai |
| 连接地址 | `jdbc:postgresql://127.0.0.1:54321/kingbase?currentSchema=biz_factor` |

#### 数据流完整链路说明

**Step 1 — AKShare 数据源 → Python 同步脚本**

Python 同步模块 `akshare-factor-sync` 调用 AKShare SDK（`ak.fund_etf_hist_em()`、`ak.fund_open_fund_info_em()` 等接口），从东方财富等公开数据源抓取基金行情、净值和标的信息。该环节对数据库**无操作**，仅读取外部 API。

**Step 2 — Python 同步脚本 → Kingbase 基础因子表**

同步脚本通过 SQLAlchemy (`utils/db_helper.py`) 连接 Kingbase，将抓取的数据写入以下原始数据表：
- `ak_fund_profile` — 基金基本信息
- `ak_fund_nav` — 基金净值时间序列
- `ak_market_quote` — 市场行情（日频 OHLCV）
- `ak_index_quote` — 指数行情
- `ak_macro_indicator` — 宏观指标
- `ak_source_task` / `ak_sync_audit` — 同步任务与审计

> **写入方式**：`INSERT ... ON CONFLICT ... DO UPDATE`（幂等 upsert），**仅追加写入，极少修改删除**。
> **写入者**：Python 进程；**读取者**：Python 计算引擎、Java 后端。

**Step 3 — Python 因子计算引擎**

`sync/derived_factors.py` 从 `ak_market_quote` 行情表读取 OHLCV 数据，计算 26 个量价衍生因子（日收益率、5/20/60 日累计收益率、移动均价、价格振幅、均换手率、量比、动量因子、反转因子、RSI、波动率、最大回撤、Amihud 非流动性等），将结果写入：
- `derivative_factor` — 衍生因子定义
- `derivative_factor_value` — 计算结果时间序列

> **写入方式**：先 `DELETE` 旧数据再批量 `INSERT`（定时全量重算）。
> **权限**：Python 端**读写**衍生因子表；Java 后端**只读**计算结果供查询。

**Step 4 — SpringBoot 后端**

Java 后端（Spring Boot 3.3.2）通过 JPA Repository 或 InMemory Repository 提供 REST API。后端主要职责：
- **元数据配置**：创建/更新/删除基础因子、衍生因子、风格因子的定义（CRUD）
- **业务查询**：读取因子值、基金信息、分类树等
- **认证鉴权**：用户登录、角色权限校验

> **读写特性**：元数据表（base_factor / factor_category / derivative_factor / style_factor）— 后端**读写**；
> 数据值表（base_factor_value / derivative_factor_value / style_factor_value）— 后端**只读**查询。

**Step 5 — Vue 前端页面**

前端（Vite + Vue 3）通过 HTTP 代理访问后端 API，实现：
- 因子树浏览与分类导航
- 基础因子与衍生因子展示
- 风格因子组合配置界面
- 因子效能分析面板
- 交易银子账户管理

> **读写特性**：纯前端展示层，通过后端 API 间接读写数据库，**不直连数据库**。

#### 各环节数据库权限矩阵

| 环节 | 语言/框架 | 基础原始数据表 | 因子定义表 | 因子结果表 | 账号角色表 |
|------|----------|:------------:|:--------:|:--------:|:--------:|
| AKShare 数据源 | Python SDK | — | — | — | — |
| Python 同步脚本 | Python / SQLAlchemy | **写** | — | — | — |
| Python 因子计算引擎 | Python / psycopg2 | 读 | 读 | **读写** | — |
| SpringBoot 后端 | Java / JPA + REST | 读 | **读写** | 读 | **读写** |
| Vue 前端 | JavaScript | — | — | — | — |

### 2.2 数据库表设计

回答"表有哪些类型"的核心问题，按业务属性将所有表分为 **4 大类型**：

| 表类型 | 核心特征 | 数据来源 | 读写特性 |
|--------|---------|---------|---------|
| **基础原始数据表** | 存储 AKShare 抓取的原生行情、财务、标的信息 | 外部数据源同步（akshare） | 追加写入，极少修改删除 |
| **因子定义与元数据表** | 存储基础因子、衍生因子、风格因子的定义和分类配置 | 系统初始化 + 用户配置 | 后端 CRUD，Python 只读 |

| **认证与权限表** | 存储系统用户账号、角色和权限模板 | 系统初始化 + 管理员配置 | 后端 CRUD，极少修改删除 |

### 4.2 核心关系拆解

#### 关系一：账号与角色（N:1）
- **关联字段**：`account.role_id` → `role.id`
- **说明**：一个角色可被多个账号引用。禁止删除被引用的内置角色。
- **业务规则**：`role.built_in = TRUE` 的角色不可删除

#### 关系二：因子分类树（自引用 1:N）
- **关联字段**：`factor_category.parent_id` → `factor_category.id`
- **说明**：支持无限级分类（当前为 2 级），通过 `cat_level` 标记层级深度
- **业务规则**：删除父分类需确认无子分类引用

#### 关系三：基础因子 → 分类（N:1）
- **关联字段**：`base_factor.category_id` → `factor_category.id`
- **说明**：每个基础因子隶属于一个叶子分类节点

#### 关系四：衍生因子 → 基础因子（N:M，通过中间表）
- **中间表**：`derivative_factor_item`
- **关联字段**：`derivative_factor_item.derivative_factor_id` → `derivative_factor.id`
- **关联字段**：`derivative_factor_item.base_factor_id` → `base_factor.id`
- **说明**：一个衍生因子由多个基础因子按权重组合；一个基础因子可被多个衍生因子引用
- **业务规则**：权重之和必须等于 100%（Java 端 `validateWeight` 校验）

#### 关系五：风格因子 → 衍生因子（N:M，通过中间表）
- **中间表**：`style_factor_item`
- **关联字段**：`style_factor_item.style_factor_id` → `style_factor.id`
- **关联字段**：`style_factor_item.derivative_factor_id` → `derivative_factor.id`
- **说明**：一个风格因子由多个衍生因子按权重组合

#### 关系六：因子值 → 因子定义（N:1）
- 三张因子值表（`base_factor_value` / `derivative_factor_value` / `style_factor_value`）分别通过 `*_factor_id` 引用各自的定义表
- **说明**：每只基金在每个时间点的每个因子取值构成一条记录
- **级联策略**：删除因子定义时，级联删除其所有因子值
